Getting Started with Che

Step by step instructions for new users. Che is supported on Windows, MacOS, and Linux. You may optionally run Che as a Docker container.

You can run Che natively (Tomcat) or in a container (Docker). Che installs with utilties to run both configurations. If you are experienced with Docker, you also launch Che from our DockerHub image.

How to Get Help

  • Issues: We want you to have a phenomenal experience with Che. These instructions should work every time, but unexpected issues do happen. You can post issues to our GitHub page. When posting, please follow the guidelines on issue reporting.

  • Documentation: We put a lot of effort into maintaining our documentation. Please check the docs to see if your issue has an explanation. If there are missing docs, errors, or areas that need more elaboration, we'd love to get that feedback.community page.

  • Chat: Che engineers hang out on Gitter, IRC and Slack. Contact details are on our community page.

Avoid These Common Setup Gotchas

Our default installer tries to give you a perfect out of the box experience. Unexpected issues can prevent this, especially on Windows. There are some simple steps you can take to avoid common issues.
  • Windows/Mac: VT-X/AMD-v must be enabled. Certain laptop vendors disable this setting by default. Che will provide you a VM error message when Che attempts to create a docker-machine VM if this is not properly set. You can change this in the BIOS. You can test to see if virtualization is enabled on Windows by running Task Manager > Performance. There will be a field labeled 'virtualization'.

  • Windows: VirtualBox can be finnicky on Windows. Our .exe installer installs VirtualBox for you. However, VirtualBox may still be unable to create a VM. The most common reason is NDIS driver bugs. You solve this by re-installing VirtualBox.

  • Windows: Avoid installing Che into Program Files. In some enterprises there are admin permissions that interrupt Tomcat.

  • Windows: If you install Che into a directory with spaces, we convert the directory to its Windows short form. This usually works, but some Windows mounted drives prevent this. Tomcat will fail as it has bugs that prevent it from resolving a CLASSPATH with spaces.

  • Mac: Only install Che into /Users directory. This is due to a limitation of virtualbox on MacOSX. We need your project files and workspaces to be saved within /Users so that virtualbox can properly mount those files into the VM. You can install Che in other directories if you choose to the use the Fusion VM driver or if you tell Che to place the workspaces and projects into /Users by modifying your che.properties to alter the project storage location.

  • Linux: Do not run Che as root. Che must be run with UID=1000. If you run Che as root, workspace creation will fail. Your projects are created by the Che process and then mounted into your workspace. In the workspace, the user is UID=1000. This user does not inherit root privileges and unable to read / write the files.

  • Networking: If you want to host Che for multiple users, you must change the networking configuration.

1. Install Che

  • Windows: Prerequisites are Java and Docker Toolbox, which installs VirtualBox, Git Bash, and Docker Machine. We provide a wizard-based installer that installs Che and its prerequisites. To install the pre-requisites manually, see our pre-requisites page. You can also install Che from signed ZIP file hosted on Eclipse mirrors.

  • MacOS: You must first install Java and Docker Toolbox. Then install Che from a ZIP file.

  • Linux: You must first install Java. Docker is installed on most Linux distributions. Then install Che from a ZIP file.

2. Run Che

  • Natively: This launches the Che server and you access it through your browser over localhost. Che's startup utilities check dependencies and start Docker (which may also start a VM). Please follow the usage steps.

  • In a Docker Container: You can optionally run Che as a Docker container. The configuration is more challenging, but you can launch the container anywhere. You do not need Che installed to run this configuration. However, Che's scripts automate automate this configuration. We have syntax for using Che's scripts, or you can launch Che with Docker syntax.

  • Success: If Che launches successfully, `http://localhost:8080` will present the dashboard for managing workspaces and projects in your browser.

3. Create Workspaces and Projects

  • Create a Workspace: Che provides a step-by-step wizard for creating your first workspace. It will pull from Che's all-in-one runtime stacks and then populate the workspace with projects that are ready to be compiled and run. We recommend that you start with the Java and Node templates.

  • Problems: If the workspace creation fails, it's usually due to some sort of network configuration issue. There are multiple connections that exist between your browser and Che; Che and the Docker daemon; and between Che and the workspace runtime. We attempt to detect any networking issue when we can and provide helpful error messages. However, you may need to review the steps that Che takes to make each connection to adjust them to your network.

  • Blank Projects: The wizard is there to make sure that new projects are matched up with a runtime stack that will have the utilities necessary to edit, build and run your project. Many projects do not fit into an all-in-one stack of technology. As an alternative step, you can create a workspace with a blank runtime in the dashboard, import projects from within the IDE, and use the terminal to install additional tools.

  • Tutorials: We provide step-by-step-by-step tutorials for getting started with Che using different technologies such as Java, Wordpress, Node.js, Subversion, and many more.

4. Setup Hosted Che

  • Multiple Users: You can set up Che to run as a server accessible by multiple users. You will need to review the steps that Che takes to configure its networking, as you will need to ensure that each user's browser can make connections to Che and the workspaces that Che is managing.

5. Customize Che

  • Plug-Ins: Tool teams can create their own plug-ins to Che which will modify the server's behavior, alter the behavior of the IDE, or inject agents into the hosted workspaces. The most common activity is to add a new project type to Che, but plug-ins have been used to change the editor, add in version control features, and to implement vendor workflows. We have instructions for writing plug-ins and extensions.

  • Assemblies: You can create your own Che packages that you distribute. The package can be run as a Che server or as a desktop IDE. You can include your custom Che plug-ins or modify Che's core system and then include these changes into your assembly. You can create custom assemblies with the included SDK tools.

6. Contribute to Che

  • Build Che From Source: We love that idea. Che is a multi-module project, so you can build all of it, or just some of it. Start in the /assembly/assembly-main/ repository fo the fastest build. We maintain build instructions on our GitHub repository.

  • Contribute: We also love this. You can contribute in many ways: documentation, writing articles, creating plug-ins, solving bugs, or getting into the core feature roadmap. We maintain a document that outlines how you contribute to Che. Also, our roadmap and weekly meeting minutes are maintained on the Che wiki.
